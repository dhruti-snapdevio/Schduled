# Solution: Pending Invites table silently swallows every action error

**Fixed:** 2026-07-16

**Files changed:**
- `components/orbit/pending-invites-table.tsx` — imported `toast` from `sonner` (already globally mounted in `app/layout.tsx`, already used for exactly this purpose in `components/orbit/audit-table.tsx`). `handleResend`/`handleRevoke` now check `"error" in result` and call `toast.error(result.error)`, or `toast.success(...)` naming the affected email on success. `handleExport` now checks `"error" in result` and calls `toast.error(result.error)` instead of silently doing nothing when the action fails.
- `lib/invitations.ts` — moved `checkInviteRateLimit` here from a private helper inside `app/actions/members.ts` (exported), matching the "extract for testability" pattern already used for `lib/roles.ts`/`lib/csv.ts`. Same behavior, just testable in isolation without importing the whole action file's dependency graph.
- `app/actions/members.ts` — now imports `checkInviteRateLimit` from `@/lib/invitations` instead of defining it locally; both call sites (`inviteMemberAction`, `resendInviteAction`) are unchanged.
- `lib/invitations.test.ts` — three new tests for `checkInviteRateLimit`: 20 allowed then blocked, invite/resend track independent buckets per actor, two actors don't share a bucket. Closes `docs/self-hosting/boss-employee-flow.md` §21.5.

**Why this fixes the root cause:** All three handlers now surface exactly what the server action reported, using the same toast mechanism already established elsewhere in the Orbit admin panel — an owner/manager gets immediate, specific feedback on every outcome (rate-limited, already-processed, or a real error) instead of a silent no-op that looked identical to success.

**How it was verified:** `tsc --noEmit` clean. `pnpm test` 56/56 (was 53 — 3 new rate-limit tests, no regressions). `pnpm build` clean, all routes compile. `pnpm db:generate`/`db:migrate` confirm zero schema drift (this fix touched no schema — expected, since it's a UI + refactor-for-testability change). Live-checked against the running dev server: signed in as a real invited manager account, confirmed `/orbit/users` renders cleanly with a real pending-invite row and no runtime or hydration errors from the new `toast` import — proving the component mounts correctly with the fix in place. The exact result shape each handler now branches on (`{error: string} | {ok: true}`) was already proven correct by the live DB-level rate-limit verification in the companion fix and by `lib/invitations.test.ts`. Visually confirming the toast itself renders on a real button click was not exercised in this pass — that needs a real browser, which wasn't available in this session; everything up to and including the data the toast receives was verified.
